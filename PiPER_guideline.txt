## Direct answer

Use **SDRE for simulation, training, calibration, checkpoint updates, and possibly remote inference**, but do **not** let SDRE directly control the physical PiPER arm.

The safe architecture is:

[
\text{PiPER + cameras} \rightarrow \textbf{local robot-control PC} \leftrightarrow \textbf{learning service / SDRE}
]

The **local robot-control PC** should be the only machine physically connected to the robot through USB-CAN and the only machine allowed to publish motion commands. SDRE can train models and return checkpoints, scores, or candidate selections, but the final command must always be checked and executed locally. This matches your CAVER proposal’s two-machine design: robot-control machine on **Ubuntu 20.04 + ROS Noetic**, learning machine on **Ubuntu 22.04 + GPU**.

The reason is practical and safety-related. PiPER’s official ROS stack is on the `noetic` branch, uses CAN activation scripts, exposes ROS services/topics for enabling, stopping, resetting, and publishing joint commands, and warns that publishing joint messages can physically move the arm. That belongs on a local machine beside the robot, not inside a queued cluster job. ([GitHub][1])

SDRE is still very useful. Public UAlberta pages describe SDRE as secure high-performance research infrastructure with NVIDIA **H200** and **L40S** GPU nodes, and your workload needs that kind of GPU capacity for OpenPI / GE-Sim / post-training. OpenPI’s own repo says inference needs more than 8 GB GPU memory, LoRA fine-tuning needs more than 22.5 GB, full fine-tuning needs more than 70 GB, and the repo is tested on Ubuntu 22.04. ([University of Alberta][2])

So the right mental model is:

> **Robot-control is local and safety-critical. Learning is remote and GPU-heavy. CAVER connects them at round boundaries or through a carefully controlled inference API, never by giving the cluster raw authority over the robot.**

# 0. The overall architecture you should build

## Recommended architecture for the first real-robot stage

Use four layers.

```text
Layer 1: Physical robot layer
  PiPER arm, gripper, cameras, calibration board/tags, table, objects, emergency stop

Layer 2: Local robot-control machine
  Ubuntu 20.04
  ROS Noetic
  piper_sdk / piper_ros
  camera drivers
  CAVER robot bridge
  hard safety shield
  runtime abort monitor
  logging

Layer 3: Learning / inference layer
  Prefer local GPU workstation if available
  Otherwise SDRE interactive GPU job or SDRE batch jobs
  openpi
  GE-Sim
  CAVER selector/calibrator
  StepNFT-style training

Layer 4: Experiment/data layer
  manifest.lock
  logs
  trajectories
  candidate chunks
  propensities
  outcomes
  model checkpoints
  SDRE storage and backup
```

The most important rule is:

> **Only the local robot-control machine may execute robot commands.**

Even if SDRE computes the candidate action, the local machine must still run:

[
S_{\mathrm{safe}}(a, s^{\mathrm{prop}}) \in {0,1}
]

before anything reaches the robot. This is exactly how the proposal defines CAVER: raw policy actions are converted by an execution adapter (\Gamma_{\mathrm{exec}}), checked by a deterministic safety shield (S_{\mathrm{safe}}), and only then executed.

# 1. Can the robot be connected directly to SDRE?

## My recommendation: no, not directly

Do **not** try to connect the PiPER arm itself to SDRE as if the cluster were the robot-control machine.

Reasons:

1. **The robot uses local hardware I/O.**
   PiPER control goes through USB-CAN / CAN interfaces. The official PiPER ROS instructions include CAN activation, CAN device naming, and ROS control nodes. Those are local hardware operations. ([GitHub][1])

2. **SDRE is a shared cluster environment.**
   Cluster jobs run through Slurm-like scheduling. General HPC guidance is that when you SSH into a cluster you land on a login node, and computational work should run through interactive or batch jobs, not directly on the login node. That model is not designed for real-time robot actuation. ([Mint Research Computing][3])

3. **Network latency and job scheduling are unsafe for servo control.**
   Your PiPER executor needs to stream commands locally at 10 Hz in the proposal. A Slurm job can be delayed, preempted, disconnected, or placed on a node that changes between jobs. That is fine for training; it is not fine for low-level robot motion.

4. **Security policy may forbid inbound/outbound robot connections.**
   I could not fetch your exact SDRE orientation page with the browsing tool, so you should confirm SDRE-specific rules with the SDRE administrators. In general, secure research environments often restrict inbound services, tunnels, exposed ports, and hardware passthrough.

## What is allowed conceptually

You can use SDRE in three safe ways.

### Mode A: SDRE for offline training only — safest and recommended first

1. The local robot-control PC runs the robot.
2. It logs trajectories, images, candidate chunks, propensities, and outcomes.
3. You upload logs to SDRE.
4. SDRE trains/refits the calibrator and updates LoRA checkpoints.
5. You download the new checkpoint to the robot-control side.
6. Next real-robot round uses the new checkpoint.

This is the best first implementation.

### Mode B: SDRE as a remote inference service — possible, but only after approval

You reserve an SDRE GPU interactively for the duration of a robot session, run a model server on that compute node, and the local robot-control machine sends observations to it and receives candidate scores or actions.

This can work only if:

* SDRE permits the network/tunnel pattern,
* the connection is stable,
* every returned action is still checked locally,
* the robot holds position or aborts on timeout,
* and the cluster is never the final actuator.

A common HPC pattern for interactive visualization is SSH tunneling from a compute node service back to a local browser; general cluster training materials describe finding the compute node and tunneling services such as TensorBoard through SSH. The same idea may be adapted for a model service, but only if SDRE permits it. ([Mint Research Computing][3])

Mode B is closer to the final proposal-mainline CAVER loop than Mode A because it preserves online provider-backed candidate scoring. But it is still a **later** path in this repo. For `F0` and the first `F1` pilot, keep Mode A as the default unless you have all of the later-readiness items in Section 13.4 and the latency evidence in Section 13.5.

### Mode C: SDRE directly controls ROS / CAN — do not do this

This is the unsafe design. Avoid it.

# 3. Safety rules before first motion

Do this before any learned policy touches the robot.

## Minimum lab rules

1. Never run robot experiments alone.
2. One person is the operator; one person watches the arm.
3. Everyone knows how to cut power.
4. Keep hands out of the workspace when the arm is enabled.
5. Start with `auto_enable:=false`.
6. Start with no learned actions.
7. Start with tiny scripted motions.
8. Use low speed and small joint deltas.
9. Log every command and every observed joint state.
10. Disable the robot when debugging software.

The official PiPER ROS README warns that publishing joint messages will make the arm lift and tells users to ensure no obstacles are in the working range. It also documents `/stop_srv` and `/reset_srv`, with reset immediately losing power and the arm falling. Treat those warnings seriously. ([GitHub][1])

## Never let multiple programs command the arm

Only one process should publish to the command topic. For CAVER, make one node authoritative:

```text
caver_robot_bridge
  receives candidate chunk
  checks safety
  streams command
  monitors feedback
  logs outcome
```

Everything else should talk to that bridge, not directly to `/joint_states` or CAN.

# 5. Connect PiPER through CAN

## 5.1 Physical connection

1. Power off the robot.
2. Connect USB-to-CAN adapter to the robot-control PC.
3. Connect CAN side to PiPER.
4. Power on the robot.
5. Keep the workspace clear.
6. Do **not** enable the arm yet.

## 5.2 Find the CAN device

From the `piper_ros` directory:

```bash
cd ~/robot_posttrain_ws/third_party/piper_ros

bash find_all_can_port.sh
```

If there is only one CAN module, the official README says you can activate `can0` at baud rate `1000000`:

```bash
bash can_activate.sh can0 1000000
```

If there are multiple USB-CAN modules, the official README gives the safer bus-info procedure:

```bash
sudo ethtool -i can0 | grep bus
```

Suppose the bus-info is `1-2:1.0`. Then activate and rename:

```bash
bash can_activate.sh can_piper 1000000 "1-2:1.0"
ifconfig can_piper
```

The official README explains this procedure and says the renamed CAN device should appear in `ifconfig`. ([GitHub][1])

Optional passive check:

```bash
candump can_piper
```

If there are no frames, do not panic immediately; some frames may appear only after launching the control node. But if the interface is not present, fix CAN first.

# 8. First motion test

Do not start with CAVER. Do not start with OpenPI. Do not start with GE-Sim.

Start with a vendor-supported small command, RViz, or a tiny scripted joint change. Keep the operator at the emergency stop.

The official README shows that publishing a `sensor_msgs/JointState` to `/joint_states` can control movement, but it explicitly warns that the arm will lift and the workspace must be clear. ([GitHub][1])

For your own tests, make a conservative wrapper instead of typing `rostopic pub` manually.

Pseudo-logic:

```python
def send_joint_target(q_target, g_target):
    q_obs = read_current_joint_state()

    assert within_joint_limits(q_target)
    assert max(abs(q_target - q_obs)) < SMALL_DELTA
    assert within_gripper_limits(g_target)

    # stream a slow interpolation from q_obs to q_target
    for q in interpolate(q_obs, q_target, steps=20):
        publish_joint_state(q, g_target)
        sleep(0.05)
```

For the first day, use:

* tiny joint deltas
* no object interaction
* no camera dependency
* no learned actions
* `auto_enable:=false` by default
* manual enable only when ready

# 9. Build the local CAVER robot bridge

Your bridge should be a local Python/ROS package. It should **not** be a quick script that directly publishes actions.

## 9.1 Required modules

Build these modules:

```text
caver_robot_bridge/
  state_reader.py
  camera_reader.py
  action_receiver.py
  safety_shield.py
  executor_10hz.py
  labeler.py
  logger.py
  experiment_state_machine.py
```

## 9.2 State machine

Use a strict state machine:

```text
DISABLED
  -> IDLE
  -> ARMED
  -> EXECUTING_CHUNK
  -> HOLD
  -> ABORT
```

Only `EXECUTING_CHUNK` may publish motion commands.

Any of these should force `ABORT` or `HOLD`:

* lost heartbeat
* missing camera frame
* missing joint state
* unsafe candidate
* SDRE/model timeout
* ROS exception
* CAN exception
* manual stop
* tracking error
* workspace violation

## 9.3 Input/output contract

The bridge should accept an execution-space chunk:

[
a \in \mathbb R^{H \times 7}
]

with rows:

[
[q_1,q_2,q_3,q_4,q_5,q_6,g]
]

In your proposal, (H=4), commands are streamed at 10 Hz, and the gripper is clipped conservatively.

The bridge should log exactly what it receives and exactly what it sends.

For every decision context, log:

```text
run_id
round_id
episode_id
context_id
task_id
language_instruction

timestamp
observed_joint_state
observed_gripper_state
camera_frame_paths
camera_intrinsics_id
camera_extrinsics_id

candidate_chunks_Hx7
safety_mask
safety_rejection_reasons
selected_candidate_index
selector_scores
propensities

executed_chunk_Hx7
execution_start_time
execution_end_time
robot_status_trace
abort_flag
label
progress_score_optional
```

Your proposal already requires this style of minimal logging for reproducibility.

# 10. Implement the local safety shield first

Do this before any policy inference.

## 10.1 Pre-execution checks

For each candidate chunk (a_{1:H}), check:

1. joint limits,
2. gripper limits,
3. velocity limits,
4. acceleration limits,
5. workspace bounds using forward kinematics,
6. table collision,
7. self-collision,
8. NaN / Inf values,
9. command discontinuity from current observed state,
10. stale robot state.

The proposal’s current safety shield uses exactly this kind of deterministic hard filter and runtime abort monitor.

## 10.2 Runtime checks

During execution, monitor at a higher frequency than command streaming, for example 20 Hz:

* observed-vs-commanded joint error
* workspace violation
* collision violation
* heartbeat
* CAN error
* `/arm_status`
* manual stop

If anything fails:

```text
stop streaming
send hold if safe
call stop service if needed
mark abort
log reason
disable robot if needed
```

## 10.3 Safety reason codes

Use a fixed enum:

```text
joint_limit
gripper_limit
velocity
acceleration
workspace
self_collision
table_collision
nan
stale_state
heartbeat
tracking
manual_estop
ros_error
can_error
```

This will be useful later for your paper, because reviewers will ask whether CAVER “works” only by rejecting many risky actions.

# 14. CAVER-specific real-robot execution loop

Once robot bring-up, cameras, safety shield, and data transfer work, the real CAVER loop is:

## 14.1 At the start of an episode

Operator resets:

```text
object placement
camera visibility
gripper state
robot home pose
workspace clear
logs started
```

The bridge records:

```text
episode_id
task_id
language instruction
template id
camera frame ids
current joint state
```

## 14.2 At a decision context

The policy proposes (K=4) candidate chunks:

[
\tilde a^\pi_{r,c,k}, \quad k=1,\ldots,4.
]

The execution adapter converts each raw chunk to an executable PiPER chunk:

[
a_{r,c,k} = \Gamma_{\mathrm{exec}}(\tilde a^\pi_{r,c,k}, s^{\mathrm{prop}}_{r,c}).
]

In your current PiPER instantiation, this means an (H\times 7) chunk with (H=4):

[
a_{r,c,k}
=========

\begin{bmatrix}
q_1 & q_2 & q_3 & q_4 & q_5 & q_6 & g
\end{bmatrix}_{1:4}.
]

The safety shield computes:

[
K^{\mathrm{safe}}_{r,c}
=======================

{k: S_{\mathrm{safe}}(a_{r,c,k},s^{\mathrm{prop}}_{r,c})=1}.
]

If no candidate is safe:

```text
send hold
log safety_abort
count context
end episode or continue according to protocol
```

If at least one candidate is safe, CAVER computes provider features, selector scores, and propensities:

[
p_{r,c,k} = \nu_r(k\mid c).
]

Then it samples:

[
A_{r,c}\sim \nu_r(\cdot\mid c).
]

The local robot-control PC revalidates (a_{r,c,A_{r,c}}), then streams it at 10 Hz.

## 14.3 After execution

The episode continues until:

```text
success
4 decision contexts used
timeout
safety abort
manual stop
```

The labeler produces:

[
Y_{r,c}\in{0,1}.
]

Then logs are stored for SDRE update.

## 14.4 After one round

After 25 counted contexts:

1. upload logs to SDRE,
2. form DR pseudo-outcomes,
3. refit calibrator,
4. update LoRA if enough positives exist,
5. download checkpoint,
6. start next round.

This roundwise design is already in the proposal.

# 22. Common mistakes to avoid

## Mistake 1: Trying to run everything on one machine

OpenPI wants Ubuntu 22.04. PiPER ROS wants ROS Noetic on Ubuntu 20.04. Keep them separate. OpenPI’s repo says it is tested on Ubuntu 22.04, while PiPER’s public ROS stack is the `noetic` branch. ([GitHub][4])

## Mistake 2: Directly publishing learned actions to `/joint_states`

Always go through the bridge and safety shield.

## Mistake 3: Using `auto_enable:=true` during debugging

Start with `auto_enable:=false`. Enable manually.

## Mistake 4: Forgetting gripper scaling

The PiPER ROS README notes RViz joint7 range and actual gripper range can differ. Your proposal uses a conservative gripper command range, but you should empirically verify units/scaling on your actual gripper before data collection. ([GitHub][1])

## Mistake 5: Letting SDRE become part of the safety loop

SDRE may compute scores, but it should not be required for immediate safety. If network/model service fails, the local robot must hold or abort.

## Mistake 6: No immutable raw logs

Never overwrite raw logs. Every result should be traceable back to raw contexts.

## Mistake 7: Starting with full CAVER

Do scripted motion, then shadow mode, then the one-task pilot, then shared seed data.

## Mistake 8: Assuming generic `/project/...` paths on SDRE

This repo is currently pinned to `/projects/p57098/euijin1/Caver` for code and `/rdss/p57098/euijin1/caver` for large artifacts.

## Mistake 9: Assuming `SCRATCH` is always usable

The environment may set `SCRATCH`, but the actual path may be absent in a given session. Verify it before using it, and prefer `/projects` or `/rdss` for anything important.
